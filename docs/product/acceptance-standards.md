# 验收标准

本文档定义 Agent Light 项目的通用验收口径，用于补充验收矩阵。验收矩阵回答“每个需求当前是什么状态”，本文档回答“什么证据才算验收通过”。

## 核心原则

1. 未读不引用，未跑不报告，未改不称已改。
2. 先按用户原话判断目标，再看 manifest 和项目证据；不能用项目现状反推用户意图。
3. 修复类任务必须先定位根因，再修改实现；不能只做症状级补丁。
4. 改函数、接口、字段、配置或协议前后，都要核对调用方、消费方和影响面。
5. 验收结论必须绑定可复核证据；证据不足时标为 `Partial`、`Unverified` 或待确认。

## 状态判定

| 状态 | 判定标准 |
| --- | --- |
| `Done` | 实现已完成，且有代码、命令输出、截图、API 响应、设备结果或人工记录等可复核证据。 |
| `Partial` | 已有部分实现或 UI 痕迹，但核心链路、异常路径、数据流或实体验证未闭合。 |
| `Unverified` | 可能已经实现，但本轮没有运行、打开、请求、构建、测试或上板验证。 |
| `Deferred` | 明确不在当前范围内，并有重新进入路线图的触发条件。 |
| `Blocked` | 目标清楚，但受权限、设备、账号、依赖、环境或外部系统阻断。 |

## 证据要求

每条验收记录至少包含：

- 需求或任务 ID。
- 验收时间。
- 验收人或执行来源。
- 使用的命令、入口、设备、接口或检查路径。
- 实际结果。
- 未验证项和剩余风险。

可接受证据包括：

- 自动化测试结果，例如 `npm run test`。
- 构建结果，例如 `npm run build` 或 Tauri 构建日志。
- API 请求和响应，例如 `GET /api/state`、`POST /api/state`、`GET /api/hardware`。
- UI 截图、录屏或手工操作记录。
- 设备侧结果，例如串口回读、固件上传日志、肉眼灯效确认。
- 代码证据，例如具体文件、函数、配置和调用方引用。

## 代码改动验收

代码改动默认需要满足：

1. 已读目标文件和直接相关上下文。
2. 已搜索关键函数、字段、状态、配置或协议的引用方。
3. 改动范围只覆盖当前目标，不夹带无关重构。
4. 能跑的测试已运行；不能跑时说明原因。
5. 核心链路、接口、状态、数据、权限、安全、存储、跨端或硬件相关改动，需要补充测试工程和代码审计视角。

纯 UI 的颜色、间距、文案、布局微调可以只做视觉验收；一旦涉及状态、接口、数据、权限、存储、关键交互或跨端行为，就恢复完整验收。

## 硬件和固件验收

硬件和固件相关任务不能只用源码存在或 API 写入判断完成。

- 固件刷写完成：必须有 `arduino-cli upload`、Arduino IDE 上传成功，或等价上传完成证据。
- 串口协议完成：必须有实际串口回读或应用侧硬件 API 返回证据。
- 灯效完成：必须有肉眼确认、照片、视频或明确的人工验收记录。
- RGB 通道诊断：优先使用纯色红、绿、蓝测试，避免混合色掩盖断线、极性或 GPIO 映射问题。

如果用户问“固件重装了吗”或“刷了吗”，回答必须直接绑定真实上传结果。API 状态写入、灯变色或源码修改不能作为刷写成功证明。

## 文档验收

文档变更需要满足：

1. 结构位置符合现有文档体系。
2. 与 spec、验收矩阵、测试策略、发布说明和运行手册不冲突。
3. 不把未验证事项写成已完成。
4. 链接、标题层级、表格和术语可读。
5. 如果文档记录测试、构建、刷写或发布结果，必须来自真实执行证据。

## 输出口径

交付时应说明：

- 命中的 must 技能和 conditional 技能。
- 关键证据。
- 改动点。
- 验证结果。
- 未验证项、阻断项和剩余风险。

没有执行的动作不能写成已执行；没有验证的结果不能写成已通过。
