# 产品路线图

## 阶段 0: 文档与基线

目标：把当前可运行 MVP 固化为 spec 模式基线。

交付物：
- MVP 产品规格
- 验收矩阵
- 架构、API、测试、发布和运行手册
- 首个 git commit

出口条件：
- 文档已覆盖当前代码能力和明确缺口。
- 测试与构建命令结果被记录。
- Deferred/Partial 项不阻塞 MVP 判断。

## 阶段 1: 本地桌面 MVP

目标：稳定交付 macOS 桌面状态伴侣。

范围：
- 主桌宠窗口
- 设置窗口
- 四状态动画与颜色语义
- 本地 API
- CLI
- Codex 状态同步
- 基础系统指标

不做：
- 真实硬件
- 小游戏完整玩法
- 自动更新
- 多平台发布

决策点：
- 开机启动是否进入 MVP。
- 是否收紧本地 API CORS。
- Codex 状态读取是否需要更稳定的内部契约。

## 阶段 2: 发布候选

目标：形成可分发 macOS `.app`。

范围：
- `npm run tauri build`
- macOS `.app` 首次启动验证
- icon、bundle id、版本号检查
- 关键链路手工回归
- 基础 release notes

暂不承诺：
- DMG
- 签名与公证
- 自动更新

## 阶段 3: ESP32 硬件适配

目标：用 ESP32 Mini + RGB 灯实现实体状态灯。

当前方案：
- USB Serial
- ESP32 Arduino 固件
- GPIO `1/2/3/G` RGB 灯

进入条件：
- 硬件协议、连接方式、电源/断连行为明确。
- 安全边界和失败兜底明确。
- 硬件 spec 和验收矩阵已更新。

后续候选：
- BLE
- 本地网络设备
- 多灯同步

## 阶段 4: 体验扩展

候选项：
- 小游戏可玩化
- 更多桌宠动作
- 自定义皮肤
- 更丰富的 agent 事件
- 用户可配置状态映射

优先级规则：
- 优先保持核心状态提醒可靠。
- 不让娱乐功能影响桌面伴侣的低干扰体验。
- 每个扩展先写 spec，再进入实现。
