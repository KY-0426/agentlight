# ADR 0001: 采用 spec 模式开发

## 状态

Accepted

## 日期

2026-06-22

## 背景

Agent Light 已经具备可运行 MVP 代码，但此前只有根 README，没有正式 spec、验收矩阵、架构、API、测试和发布文档。继续直接写代码会让范围、验收和后置项变得模糊，尤其是开机启动、小游戏、真实硬件、发布打包这些容易被误认为已完成。

## 决策

项目采用 spec 模式：

1. 任何新增功能先写或更新 spec。
2. spec 必须包含目标、范围、非目标、用户故事、验收标准、风险和依赖。
3. 实现前更新验收矩阵。
4. 实现后按测试策略记录验证。
5. 发布前按 release 文档收口。

## 影响

正向影响：

- 开发任务可以按验收标准拆分。
- Partial/Deferred 项不会混进已完成范围。
- 新人可以从文档快速理解项目。
- 后续硬件、小游戏、发布打包能独立立项。

代价：

- 每个功能进入实现前需要维护文档。
- 快速试验也要明确是否进入 spec 或仅作为 spike。

## 执行规则

- 文档和代码不一致时，以代码证据为当前事实，以 spec 为目标事实。
- 不能把未运行的命令写成已通过。
- 不能把占位 UI 写成完整功能。
- Deferred 项必须有重新进入路线图的触发条件。
- 每次较大改动后复核 `docs/product/acceptance-matrix.md`。
